---
title: Keeping reports in sync
description: Mirroring issue state back onto the report.
---

While `sync.enabled` is true, the package schedules itself — no entry in your
`routes/console.php` needed. `frequency` accepts any method on Laravel's
`Schedule` (`everyFifteenMinutes`, `daily`, …).

You can also run it by hand, or from the **Sync with GitHub** button on the list
page:

```bash
php artisan bug-reports:sync
```

Each linked issue is fetched and mirrored: a closed issue sets `resolved_at` to
the issue's `closed_at`, reopening clears it.

## Issues that no longer exist

An issue that no longer exists is skipped rather than failing the run — both the
`404` GitHub returns for an issue it won't show you (never existed, or the token
can't see it) and the `410 Gone` it returns for one that was **deleted**.

The report keeps its issue number and URL, pointing at a page that is gone. The
link is deliberately not cleared, because a deleted issue is far more often a
mistake to undo than a fact to record.

## Scheduling it yourself

Set `sync.enabled` to `false` and call `bug-reports:sync` from your own
scheduler.

## Real-time sync with webhooks

Polling closes the loop within an hour (or whatever `frequency` you set). To
mirror issue state the instant it changes, point a GitHub webhook at the package
instead — the two mechanisms are independent and can run together, the webhook
catching changes live and the schedule sweeping up anything a missed delivery
left behind.

The endpoint applies the **exact same** rule as the schedule: a `closed` issue
sets `resolved_at` to the issue's `closed_at`, `reopened` clears it. Every other
issue action is acknowledged and ignored.

### Turn it on

Set a secret — any long random string. This is the switch: while it's empty the
endpoint answers `404`, indistinguishable from a webhook that was never wired
up.

```dotenv title=".env"
BUG_REPORTS_GITHUB_WEBHOOK_SECRET=a-long-random-string
```

### Register it on GitHub

In the repository, under **Settings → Webhooks → Add webhook**:

- **Payload URL** — your app's URL followed by the configured path, e.g.
  `https://your-app.test/bug-reports/github/webhook`
- **Content type** — `application/json`
- **Secret** — the same value as `BUG_REPORTS_GITHUB_WEBHOOK_SECRET`
- **Events** — *Let me select individual events* → **Issues** only

<Callout type="warn" title="Signed, not just secret">
Every delivery must carry a valid `X-Hub-Signature-256` computed from the raw
body and your secret, or it is rejected with `403`. GitHub sends this
automatically once the secret is set on both ends; a request without a matching
signature never reaches a report.
</Callout>

### Path and middleware

The route lives at `bug-reports.webhook.path` and is named
`bug-reports.github.webhook`. It is registered **outside** the `web` group on
purpose — GitHub sends no CSRF token. Anything you add to
`bug-reports.webhook.middleware` runs *after* the signature is verified, which
is the right place for rate limiting:

```php title="config/bug-reports.php"
'webhook' => [
    'middleware' => ['throttle:60,1'],
],
```

<Callout type="info" title="Route caching">
Because the route is registered by the package, `php artisan route:cache` picks
it up like any other. Re-run it after changing the path.
</Callout>
